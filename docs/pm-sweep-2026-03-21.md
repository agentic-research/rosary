## Problem 1: Bead Inflation

**Severity: Critical**

Survey agents (dev, staging, prod, feature, janitor) create beads aggressively during
scans. There is no creation throttle or dedup at creation time. The `epic::is_dominated_by`
check prevents *dispatching* duplicates but doesn't prevent *creating* them. Result: the
backlog grows faster than the dispatch system can drain it.

Evidence: 3 exact-triplicate sets were created within a 2-hour window on 2026-03-16/17,
indicating a decompose or survey ran 3 times with no dedup guard.

**Recommended actions:**

1. Add dedup check to `rsry_bead_create` — reject if title similarity > 0.85 to existing open bead
1. Rate-limit agent bead creation per repo per session (e.g., max 20 per dispatch)
1. P0 creation by agents should require human confirmation (agents shouldn't create urgents)
